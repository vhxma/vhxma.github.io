So the whole premise of this comes from a unit I took last semester called Algorithms, Agents and Artificial Intelligence. In the unit, we covered a variety of topics including search algorithms, pattern matching, sequential decision problems (SDPs) and a whole lot more. Our main project was done in pairs where we investigated two algorithms of our choice and how effective they would be in a game of Three Chess.

Three Chess is a variant of normal chess, the main difference being that it is played with three players. The version of Three Chess that we simulated was played with a hexagonal board. Moves included castling and the standard movements for each piece.

Before I get into the nitty gritty of discussing the algorithms used in our agents, I‚Äôll first explain the underlying concepts.

The first algorithm we need to meet is called Minimax.

Minimax on a basic level is a move decision-making strategy that aims to minimise the opponent‚Äôs chance of winning and maximise your own. It assumes that the opponent will always make the most ‚Äòoptimal‚Äô moves for themselves.

How we define ‚Äòoptimal‚Äô depends on the developer.

But let‚Äôs backtrack a bit - when we play a game, we can make what‚Äôs called a ‚Äògame tree‚Äô. This is essentially a tree that will store all possible game states. For example a basic game tree in tic-tac-toe might look like this:

You can see that each level represents a board after a player‚Äôs turn. From this, you might notice how, at each level, the number of game states can start increasing drastically (it should be noted that this diagram only accounts for the next possible moves from a chosen game state in a level so is not representative of the full game tree). This is because each game has a certain set of moves and thus when we move from one level of the game tree to the next, we must account for all possible game boards that could arise from each state. Since we don‚Äôt know which ‚Äòpath‚Äô has been chosen from the initial game state, we must store all the possibilities. Hence, the number of moves that is possible from a given game state is known as the branching factor. This becomes a limiting factor in how effective Minimax can be due to the computational space limitations. Tic-tac-toe may have a small branching factor, but for games like Chess when the branching factor is a lot bigger, this can cause the game tree to grow massively.

In order to calculate what is ‚Äòoptimal‚Äô for each player, each game state is given a utility. A more negative score would mean that the game state is beneficial for the opponent, whilst a more positive score would mean it is more beneficial for you. How the utility is calculated can depend on many things like the presence of an AI who is determining what ‚Äòuseful‚Äô means, or a developer who may have implemented certain strategies. For instance, in a game of chess, a higher score may given to a game state where your pieces are in the centre.

By assuming the opponent will make the most ‚Äòoptimal‚Äô move for themselves, the computer will determine the likely route the game will take and from there, decide what is the best move. It has been proven that if Minimax is able to play out to the end of the game (when someone wins), the player can win even if the opponent does not play optimally.

Minimax is however, far from perfect. Remember how I said Minimax must play to the end of the game to be fully effective? Well, that rarely happens.

Because games usually have big branching factors, this means that Minimax must often cut off its game tree not at the end of the game. This leads to various problems such as the horizon effect (when the agent is unable to see a bad event that will happen soon i.e. A sudden change that will mean a future game state would likely lead to a loss). The shallower the depth, the more likely the Minimax agent will fail.

While there are a multitude of strategies that help improve the depth of the game tree (using symmetry in game boards, alpha-beta pruning, k-move ordering), there are still limitations to how much they can help. Consequently, for multi-player games like Three Chess, Minimax becomes unviable. For one, due to there being three players, the calculation of good and bad becomes more complex than positive and negative numbers. Secondly, the game tree that describes one round of turns suddenly takes up a lot more space than it would for a two-player game. Therefore, it takes even longer to determine a winner, and depth becomes severely limited.

So now that we‚Äôve explored the basic idea of Minimax, we will next learn about the multi-player adaptions of Minimax and how effective (or ineffective) they are.

How I Coded It

From my experience of coding this algorithm, I had to create a game tree every time my agent made a move. I did this because of the spatial requirements and difficulty of tracking which game state the game is up to. Spatial complexity aside, we now have a problem of time complexity as creating a tree also takes time.
